# Продуктовая часть

## Проблема

Во многих реальных B2B-задачах нужно прогнозировать терминальные события: отказ оборудования, уход клиента, дефолт, медицинский риск, завершение жизненного цикла договора или устройства. Это time-to-event задачи, где важны не только факт события, но и время до него, риск во времени и корректная работа с неполными наблюдениями.

Ключевая проблема: в реальных датасетах почти всегда есть цензурированные наблюдения. Для таких объектов известно, что событие не наступило до конца периода наблюдения, но неизвестно, наступит ли оно позже и когда именно. Несмотря на это, команды часто используют привычные модели классификации и регрессии.

Обычные ML-модели плохо сопоставимы между собой в такой задаче:

- классификация отвечает на вопрос "случится ли событие";
- регрессия отвечает на вопрос "когда случится событие";
- survival analysis отвечает на вопрос "как меняется риск во времени".

Классификация может принять цензурированный объект за отсутствие события и потерять информацию о времени. Регрессия часто требует точного времени события и плохо использует наблюдения, где событие не наступило. Поэтому выбор модели становится непрозрачным и потенциально некорректным.

SAWrap решает проблему выбора модели для пользовательского датасета за счет унификации прогнозов через функцию выживания `S(t | X) = P(T > t | X)`. После приведения классификации, регрессии и survival-моделей к одному survival-представлению их можно сравнивать по единым метрикам события, времени и функции выживания.

## Целевая аудитория

Open-source аудитория:

- исследователи, работающие с time-to-event данными;
- студенты и преподаватели, изучающие survival analysis;
- Data Scientists, которые хотят улучшить свои модели, сравнить разные подходы или использовать готовый pipeline;
- разработчики и ML-инженеры, которым нужны обертки для приведения классификации, регрессии и survival-моделей к единому интерфейсу.

B2B аудитория:

- компании, которым нужно выбрать лучшую модель для своего датасета;
- команды, где важно прогнозировать терминальные события: отказ оборудования, churn, дефолт, смерть, рецидив, завершение договора;
- аналитические и ML-команды, которым нужен воспроизводимый стенд для сравнения моделей на цензурированных данных.

## MVP

Текущий MVP включает:

- веб-интерфейс для выбора датасета, моделей и метрик;
- загрузку пользовательского датасета в CSV/CSV.GZ/ZIP/XLSX;
- стандартизацию датасета к виду `time`, `cens`, признаки;
- визуальное сравнение моделей на графике;
- страницу итогового ранжирования;
- предрассчитанные таблицы экспериментов;
- Docker-развертывание;
- CI и unit-тесты;
- AI-интерпретатор, который объясняет результаты;
- отдельный RAG-ассистент по проекту для вопросов по диплому, коду, таблицам и продуктовой части.

## Конкурентное отличие

Проект отличается от обычных демо ML-моделей тем, что:

- сравнивает не одну модель, а семейства моделей;
- работает с цензурированными данными;
- объединяет classification, regression и survival analysis;
- приводит прогнозы разных моделей к функции выживания;
- позволяет загрузить пользовательский датасет и выбрать модель под него;
- имеет воспроизводимый экспериментальный pipeline;
- сохраняет результаты в таблицах и показывает их через веб-интерфейс;
- использует AI для интерпретации метрик и объяснения выбора модели.

## AI-компонент

AI используется не для замены экспериментов, а для интерпретации уже рассчитанных результатов.

Текущий AI-интерпретатор:

- читает выбранный датасет и задачу;
- берет предрассчитанные метрики из `UI/tables`;
- выбирает лучшую модель по взвешенной оценке;
- объясняет, почему модель подходит;
- при наличии `OPENROUTER_API_KEY` отправляет контекст во внешнюю LLM через OpenRouter.

RAG-ассистент по проекту использует `knowledge`, диплом, README, код и результаты, чтобы отвечать на вопросы по исследованию и реализации.

## Пользовательский датасет

Пользователь может загрузить собственный датасет на сайт при запуске с включенным пересчетом:

```bash
make run SKIP_RECALC=0
```

Поддерживаемые форматы: `.csv`, `.csv.gz`, `.zip` с CSV внутри, `.xlsx`, `.xls`, `.xlsm`.

Минимальные требования:

- должна быть колонка времени наблюдения: например `time`, `event_time`, `duration`, `tenure`, `tenure_months`;
- должна быть колонка события: например `cens`, `event`, `status`, `event_observed`, `churn`, `churn_value`;
- событие кодируется как 1/true/yes, цензурирование как 0/false/no;
- после очистки должно остаться минимум 5 строк;
- в колонке события должны быть оба класса;
- время должно иметь хотя бы два разных значения;
- после удаления времени и события должен остаться хотя бы один признак.

Загрузчик приводит данные к стандартному виду `time`, `cens`, признаки, удаляет служебные и слишком шумные поля, сохраняет стандартизированный датасет в `UI/user_datasets` и позволяет запустить расчет моделей на сайте.

## Практическая ценность

Проект помогает быстрее принимать решение о выборе модели в задачах времени до события. Для open-source пользователей он снижает порог входа в survival analysis и дает готовый pipeline для экспериментов. Для B2B-сценария он превращает загрузку пользовательского датасета в понятный процесс: стандартизация данных, расчет моделей, сравнение метрик, рекомендация и демо-прогноз.

## Краткое описание

SAWrap - это open-source B2B ML-сервис для выбора лучшей модели на пользовательском time-to-event датасете. Проект объединяет классификацию, регрессию и survival analysis через функцию выживания, проводит воспроизводимые эксперименты, визуализирует результаты, использует AI/RAG для интерпретации и показывает, как Piecewise-обертки улучшают прогнозы моделей классификации за счет временной структуры.
